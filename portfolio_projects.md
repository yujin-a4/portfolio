## MAIN PROJECT 01
# 클래스캔버스 서비스 개편 기획
### "교과서 연계 쌍방향 수업 자료 1,976건을 직접 설계해 플랫폼 활용률을 끌어올리다"

### Problem

**기능은 있었다. 콘텐츠가 없었다.**

클래스캔버스의 기반인 Aspen Class는 퀴즈 제작, 쌍방향 수업, QR 공유 등 교사가 원하는 기능을 이미 갖추고 있었다. 그러나 2025년 4월 1차 오픈 당시에는 YBM 교과서와 직접 연결된 수업 자료가 없었다. 교사는 빈 도구만 받은 셈이었다.

이 문제를 데이터로 먼저 확인했다.

**Y클라우드 사용 행태 데이터 분석 결과** (활성 사용자 16,278명 기준):

- Y클라우드 교과서 자료 전체 조회의 **78%가 영어 과목에 집중**
- 초등 영어 교사의 교과서 자료 사용률 **87.4%**, 중등 영어 **80.2%**, 고등 영어 **89.4%**
- 수학, 음악, 체육, 보건 등 비영어 과목도 각각 수천 건의 조회가 발생하고 있었으나, 클래스캔버스에 연계 자료가 전무했음

**2024년 9월 Y클라우드 사용자 FGI (약 70명)** 에서는 더 직접적인 패턴이 확인됐다. 참여 교사 대다수가 쌍방향 수업 기능의 존재 자체를 모르고 있었고, 알게 된 순간 "바로 쓰겠다"는 반응을 보였다. 동시에 "수업 준비할 시간이 없어서 처음부터 자료를 만드는 건 불가능하다"는 말이 반복됐다.

> 데이터가 가리키는 방향은 하나였다.  
> **자료를 먼저 채워야 한다. 그리고 영어에만 집중하면 안 된다.**

[IMAGE: Y클라우드 사용 행태 데이터 — 과목별 교과서 자료 조회 비교 차트]

### My Role — 무엇을 기획했는가

[IMAGE: 화면 기획서 v1.0~v2.3 버전 목차 캡처]

#### 1. "YBM 자료 가져오기" 유저 플로우 설계

교사가 수업 자료를 탐색하는 경험을 처음부터 설계했다. 내 보관함에서 "YBM 자료 가져오기" 버튼을 누르면 모달 팝업이 뜨고, 그 안에서 자료를 찾아 보관함에 담을 수 있는 구조다.

핵심 판단들:

**탐색 동선 설계**: 학교급 → 과목 → 대단원/중단원 순으로 필터링했다. 교사가 수업을 준비할 때 실제로 생각하는 순서("나는 중학교 2학년 영어 3단원을 가르친다")를 그대로 따른 것이다.

**기술 제약 반영**: 과목 수가 DB 업데이트에 따라 유동적으로 변하기 때문에 고정 탭 UI로는 대응할 수 없었다. 개발사(블루가)와 협의해 스와이프 및 좌우 이동 버튼으로 대응하는 유연한 UX를 채택했다.

**유저 플로우**: 자료 선택 → 선택 카운팅 안내 → "내 보관함에 추가하기" → 완료 토스트 알럿. 섬네일 크기가 서버 측 제약으로 조정 불가능하다는 것을 개발사와 확인한 후, 그에 맞는 스크롤 영역과 반응형 레이아웃(최대 w1920 ~ 최소 w1024)을 재설계했다.

[IMAGE: YBM 자료 가져오기 화면 기획서 — 학교급/과목/단원 필터링 구조]

#### 2. 자료 품질 설계 — "수업 도구"로 만들기

교과서 내용을 PPT로 옮기면 교사 입장에서는 "이미 교과서가 있는데"가 된다. 클래스캔버스가 추가 가치를 내려면 교사 혼자서는 만들기 어려운 **쌍방향 수업 자료**를 제공해야 했다.

Aspen Class의 퀴즈 기능과 Ally Class의 쌍방향 활동이 결합된 형태로 자료가 제작되도록 가이드를 설계하고, 편집팀(초·중·고 전 학교급, 영어·수학·음악·체육·실과·보건·한문 등 다과목)을 직접 교육했다.

**과목 구성 전략**: 사용 데이터를 분석하니 영어가 조회량의 78%였지만, 수학(조회량 2위, 중등 기준 조회 78.1회/인)과 음악(비영어 과목 중 가장 높은 사용률) 역시 무시할 수 없는 수요가 있었다. 영어 자료만 구축하면 수학·음악 교사는 결국 안 쓴다는 판단 아래, 다과목 동시 구축 전략을 채택했다.

현재 구축 현황 (2026년 4월 기준):

| 과목 | 자료 수 |
|---|---|
| 수학 (초등 3~6학년) | 942건 |
| 음악 (초·중·고) | 371건 |
| 체육 (초·중·고) | 311건 |
| 영어 (초·중·고) | 195건 |
| 실과·한문·보건 | 157건 |
| **합계** | **1,976건** |

[IMAGE: 현재 클래스캔버스 내 자료 분류 화면 또는 DB 구성 시각화]

#### 3. 편집창 UX 개선

교사가 자료를 가져온 후 편집하는 과정의 사용성도 개선했다. 편집창 메뉴 구조를 재설계하여 퀴즈 템플릿·YBM 자료·갤러리 접근성을 높이고, 저장 버튼 UI 추가, OX 퀴즈 가독성 개선, QR 공유 기능을 추가했다.

#### 4. 폴더 시스템 도입

교사가 가져온 자료와 직접 만든 자료가 섞이면 관리가 어려워지는 문제를 해결하기 위해, 폴더 생성·이동·삭제 기능을 설계했다. 최대 12개 폴더, 반응형 팝업창(최대 w1920 ~ 최소 w1024)까지 고려했다.

#### 5. 외부 개발사 협업

화면 기획서를 작성하고 버전 관리(v1.0 → v1.2 → v2.0 → v2.1 → v2.2 → v2.3)를 통해 변경 이력을 추적했다. 기술 제약(섬네일 크기 조정 불가, 스와이프 구현 여부 등)이 확인될 때마다 기획을 수정하여 재전달하는 반복 사이클을 거쳤다.

**방학(1월) 딥유즈 확인** — 활성 사용자가 137명으로 줄었지만 사용자당 조회수 24.3회, 평균 참여 시간 6분 47초로 연중 최고치를 기록했다. 수업 준비를 미리 하는 코어 유저층이 존재한다는 검증이다.

신학기인 3월에는 활성 사용자 296명(10월 대비 +74%), 참여율 87.32%로 최고치를 달성했다. 별도의 마케팅 없이 서비스 가치만으로 성장한 수치다.

### Takeaway

"도구를 주는 것"과 "도구 + 콘텐츠를 주는 것"의 차이를 수치로 확인한 프로젝트다. 기능이 아무리 좋아도, 사용자가 즉시 가치를 체감할 수 있는 자료가 없으면 활용되지 않는다. 이 원리는 에듀테크뿐 아니라 모든 AI 서비스에 적용된다.

기획 역량 측면에서는, 기술 제약을 기획에 반영하는 반복 사이클이 실무에서 어떻게 돌아가는지 직접 체득했다. 기획서는 한 번 쓰고 끝나는 문서가 아니라, 개발 과정에서 계속 업데이트되는 살아있는 문서라는 것을 v2.3까지 오며 몸으로 익혔다.

## MAIN PROJECT 02
# AI 디지털교과서 — 사용자 리서치 기반 서비스 개선 기획
### "229명 교사의 목소리를 데이터로 정리하고, 설계안으로 만들어 개발사에 전달하다"

| | |
|---|---|
| **기간** | 2024.05 ~ 현재 |
| **소속** | YBM AI Lab |
| **역할** | FGI 설계·수행·분석, 품질관리 체계 설계, 개선안 기획 및 개발사 커뮤니케이션 |
| **제품** | AI 디지털교과서 — LKT 기반 학습진단, AI 보조교사, AI 튜터, 음성인식 탑재 클라우드 교육 플랫폼 |
| **규모** | 초·중·고 영어·수학 전 과목 / 보조출원사 다수 / 검정 심사 30개+ 항목 |

---

### Problem

YBM이 교육부 검정 심사에 AI 디지털교과서를 출원했다. 이 제품은 단순 전자 교과서가 아니라 LKT 기반 학습진단, AI 보조교사(실시간 수업 모니터링), AI 튜터(자기주도 학습 지원), 음성인식 기반 말하기 연습 등 다수의 AI 기능이 탑재된 클라우드 기반 교육 플랫폼이었다.

두 가지 과제가 동시에 있었다. 하나는 30개 이상의 검정 심사 항목을 통과하는 것, 다른 하나는 실제 교사가 수업 현장에서 이 서비스를 어떻게 받아들이는지를 사전에 파악하고 제품에 반영하는 것이었다.

---

### My Role

**이 프로젝트에서 수행한 업무는 크게 네 가지 축이다.**

[IMAGE: 4개 업무 축 인포그래픽 — FGI / 품질관리 / 접근성 / 자체기술검증]

#### ① 대규모 교사 FGI 기획·수행·분석 (핵심)

2024년과 2025년, 두 차례에 걸쳐 총 229명의 교사를 대상으로 FGI를 직접 기획하고 수행했다.

**질문지 설계 원칙**: 단순 만족도 조사로는 "문제가 콘텐츠에 있는지, 인터페이스에 있는지"를 구별할 수 없다. 그래서 평가 영역을 **내용적 측면 / 기능적 측면**으로 분리하고, **만족도 / 필요성**을 별도 항목으로 설계했다.

같은 모니터링 기능에 대해서도 "제공되는 정보가 수업 준비에 도움이 되는가"(내용)와 "데이터 표시 형태가 이해하기 쉬운가"(기능)를 따로 물어야 원인을 특정할 수 있기 때문이다.

[IMAGE: FGI 체크리스트 실제 문서 — 영역 구조 보이는 페이지]

---

**1차 FGI (2024년 5월 / 검정 심사 전)**

- 대상: 전국 6개 지역 현직 초등교사 157명 (영어 77명 / 수학 80명)
- 방식: 지역별 소그룹 면접 (전국 순회)
- 설문 구조: 5개 영역 × 내용/기능 2개 측면

**핵심 발견 3가지:**

| 발견 | 데이터 | 원인 분석 |
|---|---|---|
| 수업 화면 적절성 최하위 | 긍정응답률 59.9% (영어 42%) | 태블릿PC 비율 33.8%인데 최적화 미흡 |
| 모니터링 기능 인지 실패 | 사용 편의성 긍정응답률 32.4% (전체 최하위) | 기능이 다른 탭에 숨어있어 존재 자체를 모름 |
| 학생 간 상호작용 부재 | 긍정응답률 29.4% (수학 23.8%) | 영어는 협업 활동 존재, 수학은 전무 |

추가로 확인된 것: "수업 중 학생 활동 모니터링" 기능이 꼭 필요하다는 응답이 157명 중 150명. 이 데이터가 이후 개선 제안의 핵심 근거가 됐다.

**분석 → 개선 제안서 작성 → 개발사 전달**

FGI 데이터를 기반으로 "초등 AIDT 개선 제안" 문서를 직접 작성해 개발사(에누마)에 전달했다. "이거 고쳐주세요"가 아니라 설계 방향과 근거 데이터를 함께 제시하는 수준으로 구체화했다.

예시 1 — **수업안 재구성 기능 발견성 문제**: FGI에서 "기능이 있는지조차 모른다"는 발견 → 아이콘을 [수업안 만들기] 텍스트 버튼으로 변경, 저장 조건 설계, 복사·검색·공유 기능 3종 추가 설계까지 전달

예시 2 — **수학 학생-학생 상호작용 부재**: 긍정응답률 23.8%, 디지털교과서 가이드라인 필수 요소 미충족 → 곱셈 빙고, 분수 국기 활동 등 모둠 활동 6종의 규칙·진행 방식·디지털 구현 형태까지 직접 설계해 전달

모든 제안 항목에 우선순위 분류: **[요청] 제출 전 반영 / [검토] 제출 후 수정**

**2차 FGI (2025년 5월 / 운영 단계)**

- 대상: 초·중학교 교사 72명 (초등 영어·수학 / 중학 영어·수학)
- 방식: 소그룹 간담회

1차와 의도적으로 다르게 설계한 두 가지:

| 변경 사항 | 이유 |
|---|---|
| 평가 영역 5개 → 6개 + 총 50개+ 문항으로 확대 | 1차에서 "수업 이후 과정"과 "학생 관점" 데이터가 부족했음 |
| "자유 의견" 중 "기능 제안" 항목을 별도 컬럼으로 분리 | 1차에서 정성 의견이 미분류 상태로 섞여 분석 시 재분류에 과도한 시간 소요 |

**2차 핵심 발견:**

- 초등 영어: AI 보조교사 모니터링 긍정응답률 63%, 학생 개별 응답 실시간 확인 여전히 어려움
- 중등 영어: 추가 활동 탐색 효율성 긍정응답률 65%, 문항 수·소요시간·난이도 표시 요청
- 공통: "교사 판서가 학생 화면에 실시간 공유되지 않아 종이를 준비해야 하는 아이러니"

[IMAGE: 2차 FGI 결과 보고서 — 핵심 발견 수치 페이지]

---

#### ② 서비스 품질관리 체계 직접 설계 (10-3장)

검정 심사 30개 항목 중 '품질관리' 영역(10-3장)은 개발사가 아닌 내가 처음부터 직접 작성했다.

가장 중요한 판단: **시스템 품질관리와 콘텐츠 품질관리를 구분하는 것**. AI 디지털교과서는 플랫폼(시스템)과 교과서 콘텐츠가 공존하는데, 각각의 품질 기준과 관리 주기가 다르기 때문이다.

정의한 품질 지표: 시스템 가용률 99.5% 이상, 평균 응답속도 3초 이내, 콘텐츠 오류율 0.1% 이하, 사용자 피드백 반영률 80% 이상. 이슈 등급 분류(Critical/Major/Minor), 조직·역할 분장, 피드백 수렴 체계까지 포함해 운영 가능한 수준으로 구축했다.

#### ③ 자체기술검증결과서 전체 관리 (30개+ 항목)

복수 개발사가 작성한 초안을 전 항목에 걸쳐 검토·수정·보완했다. A사는 부하테스트 결과를 그래프로만, B사는 텍스트로만 서술한 것처럼 회사마다 작성 방식이 달랐다. 심사위원이 일관된 기준으로 평가할 수 있도록 서술 구조와 증빙 형식을 통일하는 작업을 했다.

#### ④ 접근성 테스트 기획·수행

시각·청각·운동·인지 장애 유형별 테스트 항목을 설계하고 직접 수행했다. KWCAG 2.1 기준에 맞는 대체텍스트 작성 기준을 수립하고 실제 콘텐츠에 적용했다. 결과를 각 개발사에 전달해 제품 개선에 반영했다.

---

### Result

- AI 디지털교과서 교육부 검정 심사 통과에 기여
- 총 229명 교사(1차 157명 + 2차 72명) FGI 기획·수행·분석 완료
- 30개+ 심사 항목 자체기술검증결과서 품질 일관성 확보
- 품질관리 체계 처음부터 설계 → 운영 가능 수준 구축
- 데이터 기반 개선 제안서 작성 및 개발사 전달

---

### Takeaway

"사용자 리서치는 질문 설계가 전부다." 1차 FGI에서 내용/기능을 분리하지 않았다면, "모니터링 기능이 다른 탭에 숨어있어 존재 자체를 모른다"는 발견은 나오지 않았을 것이다. 2차에서 기능 제안 항목을 별도 분리한 것도 1차 경험의 반성에서 나왔다.

개선 제안은 "고쳐주세요"가 아니라 "이렇게, 이 이유로"여야 개발사가 움직인다. 이 프로젝트에서 **데이터 → 인사이트 → 구체적 설계안 → 개발사 전달**의 기획 사이클을 체득했다.

## MAIN PROJECT 03
# AI Trend Lab — 사내 AI 트렌드 아카이브 플랫폼
### "구글 시트 한계를 발견하고, 직접 서비스로 만들어 팀 정식 프로젝트로 승격시키다"

| | |
|---|---|
| **기간** | 2025.11 ~ 현재 (운영 및 고도화 중) |
| **소속** | YBM AI Lab |
| **역할** | 기획 + 개발 (1인 프로젝트) |
| **기술 스택** | Next.js, React, Firebase, Vercel / 바이브코딩(Cursor + Claude + Gemini) |
| **URL** | https://ai-trend-lab.vercel.app/ |

[IMAGE: AI Trend Lab 메인 화면 캡처 — 대시보드 또는 뉴스 아카이브 화면]

---

### Problem

팀에서 매일 아침 구글 시트에 AI 뉴스 링크와 설명을 수기로 기록하며 트렌드를 조사하고 있었다. 데이터가 쌓일수록 세 가지 문제가 선명해졌다.

**탐색·검색의 한계** — 행이 늘어날수록 원하는 정보를 찾기 어려워진다. 한 달 전에 본 논문 관련 뉴스를 찾으려면 스크롤을 끝없이 내려야 했다.

**트렌드 파악 불가** — 개별 뉴스는 기록되지만, "지난 3개월간 LLM 분야에서 어떤 모델이 올라오고 내려갔는지"를 한눈에 볼 수 없었다.

**AI 리터러시 편차** — 팀원마다 관심 분야와 지식 수준이 달라, 전체가 AI 기술 흐름을 공유할 구조가 없었다.

"이걸 더 잘 할 수 있는 방법이 없을까?"라는 질문에서 출발해, 구글 시트를 대체할 사내 플랫폼을 직접 기획·개발하기로 했다.

---

### Solution — 주요 기능 4가지

[IMAGE: AI Trend Lab 기능별 화면 — 대시보드 / 뉴스 아카이브 / 순위 리포트]

#### 1. 대시보드
분야별(LLM, 이미지, 영상, 오디오, STT, 코딩 등) 주목할 만한 AI 서비스를 한눈에 조망하는 메인 화면.

#### 2. AI 뉴스 아카이브 (현재 약 150건 축적)
URL을 입력하면 Gemini API가 자동으로 제목·날짜·한줄 요약·핵심 내용 3가지·인사이트로 구조화하고, 사용자가 편집 후 게시하는 방식.

현재는 Claude 확장 프로그램을 활용해 뉴스 추가 자동화 파이프라인도 구현 중이다. 검색에는 유의어 처리를 설계해, "지피티"를 검색하면 "GPT" 결과도 함께 노출되도록 했다.

유저 플로우도 두 가지 상황으로 분기했다. 뉴스를 이미 아는 사용자는 URL 입력란에 바로 붙여넣고, 어디서 뉴스를 찾을지 모르는 사용자는 하단에 분류별로 정리된 뉴스 사이트 목록에서 둘러볼 수 있게 했다.

#### 3. 순위 리포트 — 평가 기준 설계 + 보고서 자동화

이 프로젝트에서 기획적으로 가장 깊이 고민한 부분이다.

**평가 기준 설계**: 관련 논문과 벤치마크 사이트를 리서치해 영역별 기준을 직접 수립했다. LLM은 지시이행, 코딩, 수학적 능력, 어려운 프롬프트 처리 등. 영상 생성은 역동성, 개체 유지력 등.

**보고서 자동화**: 관리자 페이지에서 벤치마크 데이터를 붙여넣기하면 AI가 정해진 양식으로 보고서를 자동 생성하도록 프롬프트를 설계했다. 현재는 AA API를 활용해 순위 데이터도 자동으로 가져오는 구조로 고도화됐다.

**가장 의미 있는 판단**: 월별 보고서가 독립적으로만 존재하면 흐름 파악이 어렵다는 점에 착안, 보고서가 쌓일수록 모델별 순위 변화를 선 그래프로 추적할 수 있도록 설계했다. 단발성 보고서를 살아있는 트렌드 데이터로 확장시킨 것이다.

[IMAGE: 순위 리포트 화면 — 선 그래프로 모델별 순위 변화 추적]

#### 4. 뉴스 리포트
주간/월간 단위 AI 뉴스 분석 리포트 정기 발행.

---

### 기획 과정의 핵심 판단들

| 상황 | 선택 | 이유 |
|---|---|---|
| 뉴스 자동 크롤링 시도 → 사이트 차단으로 실패 | 수동 URL 입력 + 크롤링 참고용 제공의 절충안 | 완전 자동화보다 "사람이 판단하고 AI가 구조화"하는 분업이 품질 측면에서 적합 |
| 월별 보고서가 독립적으로 존재 | 누적 시 선 그래프로 순위 변화 자동 추적 | 개별 스냅샷이 아닌 시계열 트렌드가 실제 의사결정에 유용 |
| 단순 키워드 검색 | 유의어 처리 설계 | AI 분야는 동일 기술에 한글명·영어명·약어 등 복수 명칭이 공존 |

---

### Result

- 자발적으로 기획·개발한 프로젝트가 팀 정식 프로젝트로 승격
- AI Lab 팀원 약 30명 대상 서비스 안정적 운영 중
- 뉴스 아카이브 약 150건 축적 / 뉴스·순위 리포트 정기 발행
- 전사 확대 배포 준비 중 (오류 수정 및 안정화 단계)

---

### 개발 과정에서 배운 것

바이브코딩(Cursor + Claude + Gemini)으로 Next.js, React, Firebase, Vercel 기반의 풀스택 서비스를 기획부터 배포까지 1인으로 완성했다. 단순히 코드를 생성하는 것을 넘어, DB 설계, API 연동, 배포 파이프라인까지 전 과정을 직접 다루며 기술적 실현 가능성을 기획 단계에서 함께 판단할 수 있는 시야를 갖게 됐다.

이 경험이 클래스캔버스 기획에서 "섬네일 크기 조정 불가"라는 개발사 회신을 받았을 때 당황하지 않고 레이아웃을 재설계할 수 있었던 배경이다.
